## Conversation Flows

### Quick Bug Report

```
User: @manager I found a bug in the button hover state

Manager: I'll help create a bug ticket. Let me gather some details...

[Manager asks targeted questions about:]
1. Which component/feature?
2. What were you doing when it happened?
3. What did you expect vs what happened?
4. Can you share a screenshot?

[After refinement:]
Manager: Here's your spec: .temp/issues/specs/20260306_123456_button-hover-bug.md
Run `/create-issue <file>` to create it on GitHub.
```

### Story with Breakdown

```
User: @manager Create a story for adding dark mode to all components

Manager: That's a large scope. Let me check if we should break this down...

[Manager detects atomicity issue:]
Manager: This affects multiple components and will need >8 subtasks.
I recommend creating:
1. An Epic: "Implement Dark Mode Support"
2. Separate Items for each component category

Proceed with Epic creation? Or would you prefer a single story?

User: Let's do the epic

Manager: [Creates epic spec with high-level milestones, no implementation details]
```

### Interactive Refinement

```
User: @manager Bug: API call fails sometimes

Manager: Let me gather details...

Q: Which API endpoint or feature is failing?
User: The user profile update endpoint

Q: How often does it fail (always/intermittently/specific conditions)?
User: Only when uploading large images

Q: What error do you see (console logs, error message)?
User: [uploads screenshot showing 413 error]

Q: What environment (browser, OS, network)?
User: Chrome on macOS, home WiFi

Manager: Thanks! Let me verify this is atomic...
[Checks: single component, single issue, clear scope ✅]

Manager: Here's your spec. Should I add any edge cases or test criteria?
User: Add a test for file size validation

Manager: [Updates spec, saves to .temp/issues/specs/]
```

## Using Examples

1. **Read the examples** to understand expected output format
2. **Try similar prompts** with @manager to see how it handles different scenarios
3. **Compare your specs** to examples to ensure quality
4. **Iterate on format** - if examples don't match your needs, update templates

## Key Patterns

### Conciseness

- Titles: <120 chars
- Sections: <500 chars
- Subtasks: Max 8 items
- Break down if it gets too large

### Atomicity

- One clear goal per ticket
- No "and" or "all components" language
- If >8 subtasks needed, suggest separate items

### Completeness

- All type-specific sections filled
- Edge cases documented
- Test criteria clear
- Acceptance criteria testable

### GitHub Integration

- Frontmatter with all metadata
- Labels properly formatted
- Project fields mapped
- Relationships specified
