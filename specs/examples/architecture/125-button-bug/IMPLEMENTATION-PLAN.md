## 11. Notes & Decisions

### Design Decisions

**Use semantic tokens for dark mode**

- **Context:** Button hover needs different color in dark mode
- **Decision:** Use `_dark:{ bg: 'colors.primary.400' }` pattern
- **Rationale:** Semantic tokens are pre-validated for WCAG AA contrast in dark mode
- **Alternatives Considered:**
  - Hard-coded hex values (rejected: not themeable)
  - RGB with opacity (rejected: doesn't handle all backgrounds)

**PandaCSS `_dark` selector**

- **Context:** Need to apply styles only in dark mode
- **Decision:** Use PandaCSS `_dark` pseudo-selector
- **Rationale:** Consistent with existing dark mode patterns in codebase
- **Alternatives Considered:**
  - CSS variables with data attributes (rejected: not PandaCSS pattern)
  - Separate dark mode file (rejected: overkill for hover fix)

### Open Questions

None - implementation path is clear.

### Technical Debt

None introduced by this fix.

### Implementation Notes

**Color token reference:**

- Light mode hover: `colors.primary.600` (darker)
- Dark mode hover: `colors.primary.400` (lighter)

**Existing pattern in codebase:**
Check `packages/core/src/components/ui/input/config/input-recipe.ts` for similar `_dark` usage.

**Testing gotcha:**
Remember to wrap test component in `<div className="dark">` to trigger dark mode styles.
